# 인쇄 모드 / 테스트 모드 도입

현재 MathPop은 "설정 → 생성 → (풀이·채점) → 인쇄"가 한 화면에서 모두 가능한 단일 모드다.
새로 **인쇄 모드**와 **테스트 모드** 두 축으로 분리해, 테스트 모드에서는 단계 프리셋 + 타이머 + 합격 판정을 자동화한다.

## 배경

레퍼런스 표(구몬식 단계별 기준):

| 단계 | 문항수 | 합격문항수(80%) | 시험시간 |
|---|---|---|---|
| A | 160 | 128 | 30분 |
| B | 130 | 104 | 30분 |
| C | 160 | 128 | 30분 |
| D |  80 |  64 | 40분 |
| E |  80 |  64 | 40분 |
| F |  85 |  68 | 50분 |
| G |  70 |  56 | 60분 |
| H |  40 |  32 | 60분 |
| I |  50 |  40 | 온라인 60분 / 오프라인 50분 |

전 단계 합격 기준은 **80%**로 공통, 단계가 올라갈수록 문항수는 줄고 시험시간은 늘어남.

## 목표

1. **인쇄 모드**: 기존 동작 유지. 수동으로 범위·연산·셈 방식·페이지 수를 고르고 자유롭게 생성/인쇄/풀이.
2. **테스트 모드**: 단계(A~I)만 선택하면 `{range, ops, mode, 문항수, 시험시간}`이 자동 세팅. 시작 → 타이머 고정 → 종료 시 자동 채점 + 합격/불합격 판정 (80% 기준).
3. **도메인과 UI 분리 유지**: 단계 프리셋은 `src/domain/stage.ts`에 순수 데이터로 두고, UI는 이걸 참조만.

## 현재 시스템 매핑 가능성

`range × ops × mode` 조합은 대략 다음과 같다.

- 1d / 2d: `+ - × ÷`
- 3d / 4d / mix / triple: `+ -` 만

즉 A~I 9단계를 **표의 문항수·시간은 그대로 채용**하되, 각 단계의 (range, ops, mode)는 MathPop 지원 범위 안에서 다시 정의해야 한다. 표의 난이도 곡선(쉬움→어려움)을 존중하면서 현재 지원 가능한 조합을 아래처럼 제안한다 — **확정 필요**.

| 단계 | 범위 | 연산 | 셈방식 | 문항수 | 시간 | 예시 문항 |
|---|---|---|---|---|---|---|
| A | 1d | + | 가로 | 160 | 30분 | `3 + 5 = ?`, `7 + 8 = ?` |
| B | 1d | − | 가로 | 130 | 30분 | `9 − 4 = ?`, `8 − 6 = ?` |
| C | 2d | + | 세로 | 160 | 30분 | `47 + 28`, `63 + 19` (세로 배치) |
| D | 2d | − | 세로 |  80 | 40분 | `72 − 35`, `91 − 47` (세로 배치) |
| E | 1d | × | 가로 |  80 | 40분 | `6 × 7 = ?`, `9 × 4 = ?` |
| F | 1d | ÷ | 가로 |  85 | 50분 | `36 ÷ 6 = ?`, `42 ÷ 7 = ?` (나머지 없음) |
| G | 3d | + − | 세로 |  70 | 60분 | `534 + 289`, `812 − 456` (세로 배치) |
| H | 2d | × | 세로 |  40 | 60분 | `47 × 26`, `83 × 59` (세로 배치) |
| I | mix 또는 triple | + − | 세로 |  50 | 60분 | mix: `1234 + 567`, `3456 − 789` / triple: `234 + 567 − 123`, `1234 − 567 + 890` |

> 현재 `div`는 1자리까지만 지원 → F/나눗셈 단계가 1d에 머무름. 표는 난이도 곡선 기준이라 "단계가 높을수록 당연히 어려운 조합"이어야 하는데, 곱셈/나눗셈이 2자리로 확장되지 않으면 E~H의 난이도 역전 가능성 있음. → **결정 필요** 항목 참고.

### 제안 2: 한국 초등 교과과정 기준 편성

위 제안은 "현재 시스템에서 뽑을 수 있는 조합을 단순히 쉬움→어려움 순으로 나열"한 것이고, 아래는 **초등 수학 교과 진도 순서**에 맞춰 다시 배치한 안이다. 학년/학기 단위 진도와 테스트 단계가 일치하면, 학부모·학생 입장에서 "지금 학교에서 배우는 부분"을 바로 고르기 쉬움.

| 단계 | 학년·학기 | 교과 단원(요약) | 범위 | 연산 | 셈방식 | 문항수 | 시간 | 예시 문항 |
|---|---|---|---|---|---|---|---|---|
| A | 초1-1 | 9까지의 수, 덧셈과 뺄셈 (합 ≤ 9) | 1d | + | 가로 | 160 | 30분 | `3 + 5`, `2 + 6` |
| B | 초1-2 | 덧셈과 뺄셈 (합 ≤ 18, 받아올림·내림 등장) | 1d | − | 가로 | 130 | 30분 | `15 − 7`, `9 − 4` |
| C | 초2-1 | 세 자리 수, 두 자리 수의 덧셈 (받아올림) | 2d | + | 세로 | 160 | 30분 | `47 + 28`, `56 + 39` |
| D | 초2-1 | 두 자리 수의 뺄셈 (받아내림) | 2d | − | 세로 |  80 | 40분 | `72 − 35`, `91 − 47` |
| E | 초2-2 | 곱셈구구 (2~9단) | 1d | × | 가로 |  80 | 40분 | `6 × 7`, `8 × 9` |
| F | 초3-1 | 세 자리 수의 덧셈과 뺄셈 | 3d | + − | 세로 |  85 | 50분 | `534 + 289`, `812 − 456` |
| G | 초3-1 | 나눗셈 (구구단 역산, 나머지 없음) | 1d | ÷ | 가로 |  70 | 60분 | `42 ÷ 7`, `54 ÷ 9` |
| H | 초3-2 | (두 자리) × (두 자리) 곱셈 | 2d | × | 세로 |  40 | 60분 | `47 × 26`, `83 × 59` |
| I | 초4-1 | 큰 수 / 덧뺄셈 혼합 계산 | mix, triple | + − | 세로 |  50 | 60분 | mix: `1234 + 567`, `3456 − 789` / triple: `234 + 567 − 123` |

**제안 1과의 차이**:

- F와 G 순서가 바뀜: 교과 진도상 **세 자리 덧뺄셈(초3-1) → 나눗셈(초3-1 후반)** 순이라 F=3d±, G=1d÷ 로 배치.
- 나머지 매핑(A·B·C·D·E·H·I)은 제안 1과 동일.

**교과과정 정합 관점의 제약**:

- 초1의 덧셈·뺄셈은 실제로 "받아올림 유무"로 1-1/1-2가 나뉘지만, 현재 `generateProblems`는 **받아올림 제어 옵션이 없어** A(합 ≤ 9)와 B(합 10~18) 수준을 1d 안에서 구분 불가. → A/B가 모두 동일한 "1d" 범위로 떨어짐.
- 초4-2 이후 등장하는 **분수·소수·혼합계산**은 현재 시스템 범위 밖 → I 단계가 사실상 "현재 시스템의 천장"이며, 구몬식 I단계(고등 난이도)와 수준 차 존재.
- 곱셈·나눗셈이 3자리 이상으로 확장되지 않아, 초4-1의 `(세 자리) × (두 자리)` / `(세 자리) ÷ (두 자리)` 단계는 표현 불가 → I 전용으로 "큰 자리 덧뺄셈"으로 대체함.

### 제안 3: 초1~초6 전체 커버리지 확장안

제안 2는 "현재 시스템으로 당장 만들 수 있는 범위"에 맞추느라 초4-1까지만 다뤘다. 초6까지 포함하려면 **시스템 기능 확장**이 병행되어야 한다. 아래 표는 12학기(초1-1 ~ 초6-2)를 9단계에 분배한 안이다.

| 단계 | 학년·학기 | 교과 단원(요약) | 범위/유형 | 연산 | 셈방식 | 문항수 | 시간 | 예시 문항 |
|---|---|---|---|---|---|---|---|---|
| A | 초1-1 | 9까지의 수, 한 자리 덧뺄셈 (받아올림·내림 없음, 합 ≤ 9) | 1d (no-carry) | + − | 가로 | 160 | 30분 | `3 + 5`, `9 − 4` |
| B | 초1-2 ~ 초2-1 | 받아올림·내림 포함 덧뺄셈 (1d~2d) | 1d (carry), 2d | + − | 세로 | 130 | 30분 | `15 − 7`, `47 + 28`, `72 − 35` |
| C | 초2-2 ~ 초3-1 전반 | 세 자리 덧뺄셈, 곱셈구구 | 3d, 1d | + −, × | 세로/가로 | 160 | 30분 | `534 + 289`, `6 × 7` |
| D | 초3-1 후반 ~ 초3-2 | 나눗셈 기초, (두자리)×(한자리~두자리) | 1d, 2d | ×, ÷ | 세로 |  80 | 40분 | `42 ÷ 7`, `47 × 26`, `63 × 4` |
| E | 초4-1 | 큰 수, (세자리)×(두자리), (세자리)÷(두자리) | 3d×2d, 3d÷2d | ×, ÷ | 세로 |  80 | 40분 | `234 × 56`, `672 ÷ 14` *(시스템 확장 필요)* |
| F | 초4-2 | 분수·소수의 덧뺄셈 (동분모 분수, 자릿수 맞춘 소수) | 분수, 소수 | + − | 세로 |  85 | 50분 | `1/5 + 3/5`, `2.4 + 3.8` *(시스템 확장 필요)* |
| G | 초5-1 | 자연수 혼합계산, 약수·배수, 약분·통분, **이분모** 분수 덧뺄셈 | 자연수 혼합, 이분모 분수 | + − × ÷ | 혼합 |  70 | 60분 | `12 + 3 × 4`, `1/2 + 1/3` *(시스템 확장 필요)* |
| H | 초5-2 | 분수의 곱셈, 소수의 곱셈 | 분수, 소수 | × | 세로 |  40 | 60분 | `2/3 × 3/4`, `1.5 × 0.6` *(시스템 확장 필요)* |
| I | 초6-1 ~ 6-2 | 분수·소수의 나눗셈 (분수÷분수, 소수÷소수) | 분수, 소수 | ÷ | 세로 |  50 | 60분 | `3/4 ÷ 2/5`, `1.8 ÷ 0.3` *(시스템 확장 필요)* |

**제안 2와의 차이**:

- A, B, C가 학기 범위를 더 넓게 묶음 (12학기 → 9단계 압축 때문).
- D~I 전체가 교과 후반부(초3-2 이후)로 밀려 고등 학년까지 커버.
- **E 이후 모든 단계가 현재 도메인 기능의 확장 없이 구현 불가**.

**필요한 도메인 확장 목록**:

| 확장 항목 | 필요 단계 | 작업 규모 감각 |
|---|---|---|
| 1. `no-carry` 플래그 (받아올림·내림 없는 덧뺄셈 생성) | A | 작음 (generate.ts에 조건 추가) |
| 2. 세 자리 이상 곱셈·나눗셈 (`3d × 2d`, `3d ÷ 2d`) | E | 중간 (기존 패턴 확장) |
| 3. **분수 타입**: `{n, d}` 자료구조, 약분/통분, 덧뺄곱나눗 | F, G, H, I | 큼 (Problem 타입 재설계, 렌더러 추가) |
| 4. **소수 타입**: 소수 생성·정렬·사칙연산 | F, H, I | 큼 (렌더러 추가, 반올림 처리) |
| 5. **혼합계산 파서**: `(a + b) × c` 같은 괄호·우선순위 | G | 큼 (문제 표현식 트리, 입력 UI 재설계) |
| 6. 이분모 분수 (최소공배수 기반 통분) | G, H, I | 중간~큼 |

**로드맵 관점의 해석**:

- 제안 2(초1~초4-1) = 현재 코드 + 소규모 추가(받아올림 플래그, 3d 곱나눗)만 있으면 **단기 구현 가능**.
- 제안 3(초1~초6) = 분수·소수·혼합계산 엔진을 얹어야 해서 **중장기 프로젝트** 규모 (사실상 MathPop 2.0).
- 전략: 우선 제안 2로 A~I 구조를 먼저 안착시킨 뒤, 분수/소수 엔진이 추가될 때 단계 테이블을 제안 3 방식으로 재편.

## 모드 전환 UX

- 헤더 최상단(또는 설정 팝오버 최상단)에 **모드 토글 (인쇄 / 테스트)** 추가.
- **인쇄 모드**: 지금 그대로. 범위·연산·셈 방식·페이지 수·타이머 수동.
- **테스트 모드**:
  - 설정 패널에 `단계 (A~I) 드롭다운`만 노출, 나머지는 프리셋에 의해 자동 세팅되고 **입력 잠김**.
  - `새로 생성` 대신 `테스트 시작` 버튼.
  - 시작 누르면: 프리셋대로 문제 생성 → 타이머 자동 시작 → 포기/중단 없음 (혹은 "중단 시 기록 미저장" 경고).
  - 타이머 0 도달 시 자동 채점 → 결과 모달(합격 여부, 정답 개수, 정답률, 소요 시간).
  - 결과 모달에서 `다시 시도` / `단계 변경` / `인쇄 모드로` 선택지.

## 도메인 변경

```
src/domain/
  types.ts          # + AppMode = 'print' | 'test', Stage = 'A' | ... | 'I'
  stage.ts          # 신규: STAGE_PRESETS (Stage → {range, ops, mode, count, durationSec})
                    #       PASS_RATIO = 0.8
                    #       isPass(correct, total): boolean
```

`generateProblems`는 현재 `(pages, ops, range)` 기반. 테스트 모드는 **절대 문항수** 단위이므로, `count` → `pages = Math.ceil(count / PROBLEMS_PER_PAGE)`로 바꾸거나, `generateProblems`에 `count` 옵션을 직접 받게 확장.

## 상태 구조 (App.tsx)

```ts
const [appMode, setAppMode] = useState<AppMode>('print')
const [stage, setStage] = useState<Stage>('A')       // test 모드일 때만 유효
const [testRunning, setTestRunning] = useState(false) // 테스트 중
const [testResult, setTestResult] = useState<TestResult | null>(null)
```

- 인쇄 모드: 기존 상태 그대로 (mode, ops, range, pages)
- 테스트 모드: `stage` 변경 시 프리셋으로 `(mode, ops, range, pages, timerDuration)` 덮어쓰기 + 잠금
- 테스트 시작: `resetState()` + `generate()` + `startTimer()`
- 타이머 종료 이벤트: `gradeAll` 호출 → `testResult` 세팅 → 모달 오픈

## 결정 필요 항목

1. **9단계 매핑 확정**: 위 표가 그대로 OK인지, 특정 단계(F 나눗셈, H 곱셈 등)를 다른 조합으로 바꿀지.
2. **곱셈/나눗셈 범위 확장 여부**: 현재 2d까지 `mul`, 1d까지 `div`만 지원. H 단계(2d 곱셈)는 가능하지만 더 상위 단계용 3d 곱셈은 없음. 향후 확장 계획?
3. **테스트 중단 정책**: 중단 버튼 허용? 허용 시 기록 미저장으로 끝낼지, 시간 초과로 강제 채점할지.
4. **결과 기록 저장**: 로컬 저장(localStorage)해서 이력 조회 기능 추가할지, 아니면 세션 단위로만 동작하고 사라질지.
5. **테스트 모드에서 인쇄 허용 여부**: 테스트 모드도 시험지 인쇄가 필요한가? (오프라인 시험 대비)
6. **채점 UI**: 기존 "N문제 중 M개 정답 (정답률: X%, 점수환산: Y점)" 인디케이터와 통합할지, 결과 모달 전용으로 분리할지.
7. **인쇄/테스트 모드 토글 위치**: 헤더 메인 바 vs 팝오버 최상단. 모바일에서는 공간 제약 고려.

상세한 실행 단계는 [`checklist.md`](./checklist.md) 참고.
